1.2 O Problema Central
Diagnóstico: JSONB livre sem contrato de dados
• audio_classifications.classification_data é JSONB sem schema definido
• O LLM recebe a transcrição inteira e produz um JSON diferente a cada execução
• Os 9 módulos de relatório tentam parsear campos que podem ou não existir
• Resultado: relatórios genéricos, campos undefined, comparações impossíveis entre atendimentos
• Causa raiz: não há separação entre extração determinística e julgamento semântico
O schema real esperado pelos renderers foi mapeado diretamente do código. São 20 top-level keys com subestruturas específicas. Nenhuma delas tem garantia de existência no JSONB atual. A Seção 4 apresenta o schema completo derivado do código.
2. Princípio Arquitetural — Divisão de Responsabilidades
A causa dos relatórios genéricos é estrutural: o LLM está sendo usado para fazer tudo. A arquitetura correta separa o que é computável deterministicamente do que exige inferência semântica. Cada método tem seu escopo e nenhum faz o trabalho do outro.
Regra absoluta do pipeline
O LLM nunca recebe a transcrição inteira para analisar tudo de uma vez.
Cada agente LLM recebe contexto mínimo e responde UMA pergunta específica com schema obrigatório.
Tudo o que pode ser computado sem LLM, DEVE ser computado sem LLM.

3. Pipeline End-to-End — 5 Etapas
O pipeline executa sequencialmente. As etapas E1 e E2 são totalmente determinísticas e de custo zero em LLM. As etapas E3 e E4 envolvem agentes LLM com contexto controlado. E5 persiste o resultado em schema fixo.
E1
Normalização da Transcrição
Parsing puro · Zero LLM · Input: texto bruto da transcrição
A transcrição do Whisper chega como texto corrido com marcadores informais ou sem marcadores. Esta etapa converte em array de turnos estruturados — a unidade atômica de toda análise posterior.
Algoritmo de parsing de turnos
Detectar padrões de marcador explícito: 'Vendedor:', 'Cliente:', 'V:', 'C:', 'Atendente:'
Se sem marcadores: inferir por posição alternada (V, C, V, C...) com parágrafo como separador
Normalizar texto: expandir abreviações, remover ruídos, aplicar pontuação mínima
Output: [ { id, role: VENDEDOR|CLIENTE|TERCEIRO, text, char_start, char_end } ]
Score de confiança do parsing
Atribuir confidence_score ao parsing baseado na presença ou ausência de marcadores explícitos.
Se confidence < 0.6 → flag manual_review = true no registro do áudio.
Transcrições de baixa qualidade com role errado corrompem toda análise downstream.
E2
Extração Determinística
Dicionários + Regex · Zero LLM · Input: array de turnos
Com os turnos estruturados, executa-se varredura computacional para extrair tudo o que pode ser detectado com certeza. Esta etapa resolve a maior parte do checklist Zeiss e todas as métricas objetivas sem gastar um token sequer.

2c — Checklist Zeiss V.6 — Itens que Exigem LLM (7 de 22)
Itens sem evidência linguística direta → passam para Agente 1
• explicou_zeiss_com_clareza → avaliação qualitativa do conteúdo explicado, não só presença de keywords
• realizou_medicao_com_equipamento → pode não ter keyword explícita se feito sem comentário verbal
• ofertou_lente_maior_valor_primeiro → requer análise de sequência de apresentação de produtos
• demonstrou_interesse_genuino → padrão conversacional (escuta ativa, perguntas de continuidade)
• utilizou_recurso_visual → tablet/demotool pode ser usado sem menção verbal
• consultou_tabela_precos → pode ser feito silenciosamente
• contato_pos_visita_7_dias → requer dado externo ou follow-up registrado — não resolvível por transcrição
E3
Agentes de Julgamento Semântico
7 agentes LLM · Contexto mínimo · Schema obrigatório
Os agentes LLM recebem exclusivamente contexto estruturado produzido pela E2, nunca a transcrição inteira. Cada agente tem um único escopo, uma única taxonomia e um schema de saída validado por código antes de ser aceito.
Agente 1 — Julgamento de Checklist Ambíguo
Input: itens com requires_llm=true + turnos relevantes + produtos detectados na E2.
Output obrigatório por item: { verdict: SIM|NÃO|INCONCLUSIVO, confidence: 0-1, evidence: string, turn_id: int }
Agente 2 — Classificação de Objeções e Eficácia
Input: todos os turnos do CLIENTE. Taxonomia fechada de tipos de objeção:
Tipo de Objeção
Definição
PRECO_ALTO
Cliente expressa que o preço está acima do esperado
PRECO_SEM_CONTEXTO
"Vou pensar" sem objeção explícita — hesitação genérica
PRECISA_CONSULTAR_TERCEIRO
Precisa falar com cônjuge, sócio, parente
JA_TEM_PRODUTO_SIMILAR
Já tem óculos recente ou produto equivalente
COMPARACAO_CONCORRENTE
Menciona concorrente com preço ou qualidade favorável
QUALIDADE_QUESTIONADA

URGENCIA_BAIXA

Questiona parcelamento, disponibilidade de crédito, entrada
DESCONFIANCA_MARCA
Ceticismo sobre Zeiss especificamente
Output por objeção: { type, text, turn_id, vendor_response_text, efficacy: CONTORNOU|NAO_CONTORNOU|PARCIAL, phase }
Agente 3 — Scoring de Intenção de Compra
Input: turnos do CLIENTE + objeções classificadas + entidades detectadas (preços perguntados, produtos experimentados).
Usa Chain-of-Thought interno obrigatório antes de emitir score.
Output: { intent_score: 0-100, intent_classification: BROWSING|PESQUISANDO|PRONTO|URGENTE, cot_reasoning: string, urgency_detected: bool, urgency_evidence: string, buy_triggers_detected: string[] }
Agente 4 — Sentimento por Fase
Executa em loop por fase (não por transcrição inteira). Recebe: segmentos de uma fase + role do falante.
Output por fase: { phase, sentiment_label: positive|neutral|negative, intensity: 0-1, dominant_emotion: frustração|entusiasmo|dúvida|satisfação|pressa|tédio }
Output agregado: conversation_heatmap → array de segmentos com índice e intensidade para visualização.
Agente 5 — Perfil Psicológico e Comportamental
Input: todos os turnos do CLIENTE + métricas de fala da E2.
Output: { archetype: RACIONAL|EMOCIONAL|SOCIAL|CONSERVADOR, decision_style: string, risk_profile: string, trust_level_towards_seller: string, price_sensitivity: high|medium|low|uncertain, inferred_age_range: string, inferred_gender: string, inferred_socioeconomic_class: string, estimated_purchasing_power: string, purchase_type: new_customer|returning_customer|uncertain, preferred_communication_style: string, client_engagement_level: high|medium|low, client_knowledge_level: expert|intermediate|novice, repurchase_signal: bool }
Agente 6 — Oportunidades Perdidas
Input: entidades de produtos detectadas + catálogo completo + turnos do CLIENTE.
Output: { total_estimated_value_loss: number, cross_sell_not_offered: [ { product, estimated_value, client_signal } ], upsell_not_offered: [ { product_offered, product_premium, estimated_delta_value } ], implicit_needs_not_addressed: string[], products_requested_not_shown: string[], products_shown_not_requested: string[] }
Agente 7 — Síntese Qualitativa (executa por último)
Este agente recebe os outputs estruturados dos 6 agentes anteriores — nunca a transcrição. Gera narrativa a partir de fatos já classificados, não de inferências sobre texto bruto.
Output: { summary_narrative: string (3-5 parágrafos), strengths: string[], improvements: string[], training_recommendation: string, compliance_flags: { abusive_practice_detected, unfulfillable_promise_detected, warranty_mentioned, consumer_rights_mentioned, mandatory_script_followed, compliance_score, compliance_alerts: string[], abusive_practice_description, unfulfillable_promise_description }, sales_techniques_detected: string[], empathy_demonstrated: bool, active_listening_signals: bool, language_complexity: technical|balanced|simple, jargon_usage: high|medium|low, personalization_score: 0-100, follow_up: { contact_info_collected, follow_up_promised, contact_type_collected, follow_up_timeframe_mentioned }, loss_to_competitor_risk: string, competitive_mentions: [ { competitor_name, comparison_direction, client_statement } ] }
E4
Validação e Consolidação
Código puro · Zero LLM · Input: outputs dos 7 agentes
Schema validation: verificar tipos, ranges (0–1, 0–100), enums e campos obrigatórios de cada agente
Re-execução automática: máximo 2 tentativas com prompt de correção explicitando o erro de schema
Rastreabilidade: cada campo recebe source (DETERMINISTIC ou LLM_AGENT_N), evidence, turn_id, confidence
Consolidação: código monta o objeto final — o LLM nunca monta o objeto consolidado
E5
Persistência Estruturada
Supabase · Schema fixo · Relatórios via SQL
O objeto consolidado é persistido em tabelas relacionais com schema fixo. Esta é a mudança arquitetural mais crítica: relatórios quantitativos passam a ser queries SQL determinísticas, comparáveis entre atendimentos, sem dependência de LLM.

5. Plano de Execução — 5 Fases
Cada fase entrega valor independente. O sistema atual continua funcionando em paralelo durante toda a transição. Nenhuma fase bloqueia o uso do endpoint /classify existente até que a migração esteja validada.
FASE 1 — Fundação — Ontologia e Schema
Prioridade máxima · Sem ela, nenhuma outra fase tem valor
Por que começar pela ontologia, não pelo código
O schema das tabelas e os arquivos JSON de ontologia são o contrato que governa tudo.
Mudanças de schema após a Fase 2 geram retrabalho em cascata nos agentes e nos relatórios.
Invista 2–3h na revisão manual da ontologia antes de abrir o Antigravity.

8. Decisões Críticas e Riscos
8.1 Migração Gradual — Sem Big Bang
O endpoint /classify atual continua operando. O novo /analyze-v2 funciona em paralelo. A migração do relatorio_dinamico.html para as views SQL é feita módulo a módulo (r_qa_heatmap primeiro, por ser o mais crítico para o caso Zeiss). Nenhuma migração quebra o sistema em produção.
8.2 Schema Imutável na Fase 1
Os arquivos JSON de ontologia e as tabelas SQL definidos na Fase 1 não podem mudar após o início da Fase 2. Mudanças exigem migração de dados e refatoração dos agentes. Este é o único ponto no projeto onde vale gastar tempo extra antes de codar.
8.3 T1.1 é Manual — Não Delegue ao Antigravity
A revisão da ontologia do checklist Zeiss é trabalho humano
O checklist_zeiss_v6.json não pode ser gerado pelo Antigravity sem revisão manual.
Cada keyword gatilho precisa ser validada contra transcrições reais.
Um falso positivo no item 'cumprimentou_cliente' distorce o score de todos os atendimentos.
Dedique 2–3 horas revisando os 22 itens antes de prosseguir para T1.2.
8.4 Risco — Qualidade da Transcrição
Risco principal: role assignment incorreto no parser
Se a transcrição não tiver marcadores de falante, o parser usa heurística posicional.
Múltiplos vendedores, terceiros na conversa ou baixa qualidade de áudio geram role errado.
Role errado no VENDEDOR ou CLIENTE invalida todo o checklist e as métricas de fala.
Mitigação: confidence_score no parser + flag manual_review para casos abaixo de 0.6.
9. Sequência de Início Imediato
A Fase 1 começa pela ontologia — não pelo código. Os 3 arquivos JSON de ontologia são a única peça do sistema que exige conhecimento de domínio que o Antigravity não tem. Tudo o mais pode ser delegado.
Sequência para começar
1. [MANUAL 2–3h] Revisar e completar checklist_zeiss_v6.json com os 22 itens, keywords e pesos
2. [MANUAL 30min] Criar product_catalog.json com produtos e variações linguísticas do catálogo Zeiss
3. [MANUAL 15min] Criar objection_taxonomy.json com os 9 tipos, definições e exemplos
4. [ANTIGRAVITY T1.2] Criar script SQL das 6 novas tabelas seguindo o padrão de classification_setup.sql
5. [ANTIGRAVITY T1.3] Implementar parser.js + extractor.js com funções especificadas na Seção 3
6. [VALIDAÇÃO] Testar com 3 transcrições reais — verificar role assignment e detecção de checklist
7. [ANTIGRAVITY T2.1] Criar /analyze-v2 com orquestrador do pipeline
8. [ANTIGRAVITY T2.2–T2.3] Implementar os 7 agentes com schema da Seção 4
