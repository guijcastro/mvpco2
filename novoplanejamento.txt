Princípio Central
Todo processamento que puder ser feito localmente DEVE ser feito localmente. O LLM entra somente onde a análise exige julgamento qualitativo que código não produz. Cada agente LLM recebe o menor contexto possível, responde UMA pergunta com schema obrigatório validado por código. Este documento é a fusão completa do diagnóstico original (p0), da especificação dos 9 agentes com schemas completos e matriz de rastreabilidade (v2), e da auditoria item a item das 40 features com separação local/LLM por sub-etapa (v4).

1.2 Problema Central
DIAGNÓSTICO: audio_classifications.classification_data é JSONB sem schema definido. O LLM recebe a transcrição inteira e produz JSON diferente a cada execução. Os 9 módulos tentam parsear campos que podem não existir. Resultado: relatórios genéricos, campos undefined, comparações impossíveis entre atendimentos. Causa raiz: (1) não há separação entre extração determinística e julgamento semântico; (2) os agentes não são mapeados para as 40 funcionalidades do produto; (3) tudo que é possível fazer localmente é delegado ao LLM.
1.3 Princípio Arquitetural — O Erro a Evitar
O LLM não é motor de análise. É árbitro de julgamento qualitativo em situações específicas. Cada decisão de usar LLM deve ser justificada pela impossibilidade de resolver localmente. Features que podem ser resolvidas com regex, dicionário, SQL ou aritmética DEVEM ser resolvidas assim. Este plano documenta, para cada feature, qual parte resolve local e qual parte — se alguma — vai ao LLM.
